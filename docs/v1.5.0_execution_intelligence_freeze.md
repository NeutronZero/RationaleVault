# RationaleVault Architecture Freeze — v1.5.0 Execution Intelligence

This document locks the public API surface and database schemas for RationaleVault's **Execution Intelligence** subsystem (Epic C5). It establishes the formal boundaries for historical telemetry analytics, rule assessments, interpreted intelligence reports, and immutable learning feedback records.

## 1. Frozen Public API Surface

The following contracts are frozen and must not be altered:

### Analytical Contracts
* **`ExecutionAnalyticsState`**: Stable projection state wrapper that maps compiled execution metrics, containing execution input/output snapshot hashes and an `analytics_hash`.
* **`ExecutionAnalytics`**: Aggregated facts maps covering skill execution records, plugin execution stats, durations, promotion gate telemetry, and cost breakdowns.
* **`ExecutionAnalyticsConfig`**: Shared configuration settings for rolling window size, trend boundaries, cost model configurations, and weightings.

### Assessment Contracts
* **`AssessmentScoreBreakdown`**: Immutable breakdown recording metrics-based performance scores (reliability, performance, promotion, cost, and overall rating).
* **`AssessmentRuleResult`**: Record representing the result of a single assessment execution.
* **`AssessmentRule`**: Extensible interface for custom rules assessing execution quality.
* **`ExecutionAssessment`**: Collection of rule evaluations, score breakdowns, and health status/health trend classifications.
* **`HealthStatus`**: Classification states (`HEALTHY`, `WARNING`, `UNHEALTHY`).
* **`HealthTrend`**: Trend direction determinations (`IMPROVING`, `STABLE`, `DEGRADING`).

### Intelligence & Report Contracts
* **`RetryDecision`**: Enumerated recovery options (`NONE`, `IMMEDIATE`, `BACKOFF`, `MANUAL`, `PERMANENT_FAILURE`).
* **`RetryRecommendation`**: Output suggesting retry actions, mapping the originating `execution_id` and `result_id`.
* **`ExecutionFinding`**: Structural diagnostic finding representing anomalies (timeouts, validation issues).
* **`ExecutionIntelligence`**: Synthesized collection of retry decisions, cost allocations, and findings.
* **`ExecutionIntelligenceReport`**: Public immutable report snapshot representing execution telemetry.

### Feedback & Learning Contracts
* **`PlannerFeedback`**: Ephemeral interpreted planner guideline containing priority adjustment weights, profile hints, and rationale.
* **`LearningRecordBuilder`**: Subsystem compiling deterministic identifiers and persistent records.
* **`ExecutionLearningRecord`**: Immutable event-sourced record tracing feedback, versioning, assessment/analytics hashes, and source execution/artifact identifiers.
* **`LEARN-[hash]` ID format**: Hashing rule deterministically derived from `planner_feedback`, `evaluation_version`, and `assessment_hash`.

---

## 2. Frozen Projection Dependency Graph

Execution intelligence projections must conform to this strict Layer 1 (Base) to Layer 3 (Intelligence) DAG. Custom rules or feedback modules must not bypass these layers or reference backward-flowing dependencies:

```text
ExecutionStateProjection
          │
          ▼
ExecutionAnalyticsProjection
          │
          ▼
ExecutionIntelligenceProjection
```

---

## 3. Unfrozen Subsystem Configurations (Allowed to Evolve)

The following metrics, heuristics, and config settings reside on the policy side of the boundary and are allowed to evolve:
* Specific health classification thresholds (e.g. at what success percentage a skill becomes `WARNING`).
* Retry backoff interval values and retry limit bounds.
* Compute cost weights (memory/duration multipliers).
* Trend epsilon bounds.
