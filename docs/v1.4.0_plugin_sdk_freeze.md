# RationaleVault Architecture Freeze — v1.4.0 Plugin SDK

This document locks the public API surface of the RationaleVault Plugin SDK for the v1.4 release. By freezing these boundaries, we guarantee backward compatibility for third-party plugin authors while allowing the platform's core runtime activation, scanning, and loading subsystems to evolve independently.

## 1. Frozen Public API Surface

The following contracts are frozen and must not be altered:

### Data Structures & Contexts
* **`PluginContext`**: Standard runtime settings, API versions, and capability grants passed to plugins during skill creation.
* **`PluginDescriptor`**: Scanned declarative metadata (ID, version, name, entrypoint, required capabilities).
* **`PluginCapabilities`**: The specific permission scopes (`requires_network`, `requires_filesystem`, `requires_subprocess`, `experimental`) a plugin requests.
* **`SkillInput`**: The immutable, versioned, hashable value bundle consumed by skills.
* **`SkillOutput`**: The immutable, versioned, hashable value bundle produced by skills.
* **`ArtifactCandidate`**: The descriptive representation of potential artifacts output by skills.

### Base Classes
* **`BasePlugin`**: The core ABC implemented by dynamic plugin modules. Must expose `manifests() -> list[SkillManifest]` and `create(skill_id, context) -> BaseSkill`.
* **`BaseSkill`**: The core ABC implemented by extension skills. Must declare capability metadata class variables and implement `__call__(skill_input: SkillInput) -> SkillOutput`.
* **`SkillManifest`**: The declarative specification of skill schemas, inputs, outputs, categorizations, and timeouts.

---

## 2. Unfrozen Platform Internals (Allowed to Evolve)

The following components reside on the platform side of the trust boundary and are **not** frozen in v1.4.0:

* **`PluginScanner`**: Reads directories and builds descriptors. Can evolve to scan archives, query remote registries, or inspect compiled binaries.
* **`PluginLoader`**: Performs dynamic code importing. Can evolve to sandbox imports, execute WASM runtimes, run RPC calls, or spin up isolated subprocesses.
* **`PluginValidator`**: Asserts security requirements. Can evolve to support cryptographic signatures, checksum verification, or advanced static analysis.
* **`PluginRegistry`**: Tracks active loaded instances and lifecycle statuses.
* **`SkillResolver`**: Maps resolved manifests to descriptor locations.
* **`SkillActivator`**: Handles lazy-instantiation and runtime wrapper generation.

---

## 3. Trust Boundary Invariants

The platform enforces that dynamic plugins remain strictly **output and metadata producers**:
* Plugins **cannot** read or write directly to the Event Ledger.
* Plugins **cannot** directly promote artifacts, generate `SkillResult` events, compile `ExecutionReport` registries, or evaluate `GateResult` decisions.
* Plugins run within the boundaries of the platform-provided sandbox, which validates inputs, checks permissions, and terminates executions exceeding timeouts.
